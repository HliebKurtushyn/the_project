# Workflow (правила і т.д.)

## Коміти
- Використовуйте зрозумілі та описові повідомлення комітів
- Основні префікси для повідомлень комітів:
  - `feat:` - нова функціональність
  - `fix:` - виправлення помилки
  - `docs:` - зміни в документації
  - `style:` - зміни, що не впливають на логіку коду (форматування, пропуски і т.д.)
  - `refactor:` - зміни в коді, які не додають нової функціональності і не виправляють помилки. Наприклад, перейменував функцію з `add_user` на `register`.
  - `test:` - додавання або зміна тестів
  - `chore:` - зміни в процесах збірки, інструментах або бібліотеках

## Робота з гілками
- Якщо йде робота по новому функціоналу (а так ймовірно буде завжди), називайте свою гілку відповідно до функціоналу, над яким ви працюєте. Наприклад, `feature/user-authentication` або `bugfix/login-issue`.

## Pull Requests
- Перед створенням pull request переконайтеся, що ваш код працює нормально і не ламає існуючий функціонал.
- Після завершення роботи виконується:
  - `git push origin <branch-name>`
  - створюється Pull Request на GitHub (base: master).
- Після цього, тімлід переглядає ваш код, дивиться чи все ок, і якщо все добре - зливає вашу гілку в `master`.

## Стиль коду
- Будь ласка, PEP8!!! Спитайте в ШІ, він пояснить базу. Не обов'язково читати реальну документацію. Якщо коротко, це правильні відступи, коментарі й інше.
- Використовуйте зрозумілі імена змінних, функцій і класів.


## Що до ШІ
Будь ласка, не пишіть код чисто з ним. Як нам говорив вчитель - **ми повинні розуміти, що ми робимо**. Тож використовуйте ШІ як помічника, а не як основного розробника.
Я зовсім не проти ШІ, і чам ним часто користуюсь, але варто самому розуміти що ми робимо. Найкраще його використовувати для генерації **ідей, пояснень, або навіть прикладів коду**, але не для повного написання функціоналу.
Також якщо є якісь помилки на яких застрягли - ШІ може допомогти їх пояснити і навіть запропонувати рішення.

## Pull Request чеклист
- [x] Код працює і не ламає існуючий функціонал
- [x] Всі тести проходять успішно (якщо є тести)
- [x] Ви самі розумієте зміни, які вносите
- [x] Документація оновлена (якщо потрібно)
- [x] Міграції застосовані (якщо потрібно)


## Стандарти
### Шаблони
- Шаблони зберігаються в папці `templates/<назва апки>`.
- Базовий шаблон називається `base.html` і знаходиться в папці відповідної апки.
- Всі шаблони повинні наслідуватися від `base.html`:
```html
{% extends "base.html" %}
```

### Статичні файли
- Статичні файли (CSS, JS, картинки) зберігаються в папці `static/<назва апки>/`.
- Використовуйте відповідні підпапки для організації файлів:
  - CSS файли в `static/<назва апки>/css/`
  - JS файли в `static/<назва апки>/js/`
  - Картинки в `static/<назва апки>/images/`
